2. Problem Definition

The Flaws of Remote Meetings

Despite rapid advancements in software, remote workers and students continuously face severe disruptions during online meetings.

2. Problem Definition

1. Background Noise

Dogs barking, traffic, construction, and keyboard typing ruin meeting clarity and professionalism.

  • Causes fatigue for listeners.
  • Distracts from the core message.
  • Current solutions are often paid plugins or computationally heavy.

2. Problem Definition

2. Unreliable Transcriptions

Many meetings are lost to time because manual note-taking is inefficient, and built-in transcriptions are inaccurate.

  • Missing context for those who couldn't attend.
  • Lack of speaker diarization (who said what?).
  • Poor handling of different accents.

2. Problem Definition

3. Fragmented Collaboration

Users must juggle multiple apps: one for video, one for whiteboard, one for file sharing.

  • Decreases productivity.
  • Causes "context switching" fatigue.
  • Difficult to synchronize all participants on a single workflow.

2. Problem Definition

4. Poor Scalability

Legacy systems struggle with bandwidth management when participant counts increase, leading to frozen videos and robotic audio.

3. Solution

MeetVerse: The Unified Hub

An all-in-one, intelligent platform designed from the ground up to solve modern meeting fatigue.

3. Solution

1. AI Audio Shield

We implemented a dedicated Python/FastAPI microservice running an advanced PyTorch model.

  • Processes audio streams in real-time.
  • Isolates human voices and suppresses background noise automatically.

3. Solution

2. High-Fidelity Transcriptions

Integrated with the Deepgram API via secure backend controllers.

  • Incredibly fast and accurate Speech-to-Text.
  • Speaker identification included.
  • Captions rendered dynamically on the frontend.

3. Solution

3. Unified Collaboration Tools

Built-in interactive tools directly inside the meeting view.

  • Real-time synchronized Whiteboard.
  • Seamless Screen Sharing.
  • Live Chat and Hand-Raising features.

3. Solution

4. WebRTC via LiveKit

We abandoned legacy peer-to-peer mesh networks in favor of an SFU (Selective Forwarding Unit) architecture.

  • Handles hundreds of participants efficiently.
  • Dynamically scales video quality based on bandwidth.

Provides live closed captions for all speakers. Crucial for accessibility and ensuring no details are missed in noisy environments.

4. Features

Powered by React Flow / Custom Canvas, users can draw, erase, and map ideas in real-time. State is synced perfectly using ASP.NET SignalR Hubs.

5. System Analysis & Design

Clean Architecture Approach

The backend was strictly designed using Clean Architecture principles to ensure scalability, testability, and separation of concerns.

5. System Analysis & Design

The 4 Layers

Web Presentation Layer (Controllers)
Application Layer (Interfaces, DTOs, Logic)
Domain Layer (Entities, Value Objects)
Infrastructure Layer (EF Core, DbContext)

5. System Analysis & Design

Domain-Driven Design (DDD)

  • Entities encapsulate their own business logic.
  • Rich domain models prevent "anemic" classes.
  • Repositories handle data access abstraction.

5. System Analysis & Design

Database Schema Highlights

We utilized PostgreSQL via Entity Framework Core.

  • Users: Managed by ASP.NET Identity.
  • Meetings: Hosts, Passwords, Schedules, Status.
  • Recordings & Transcripts: 1-to-Many relationships for meeting assets.

5. System Analysis & Design

Real-Time Infrastructure

We deployed dual real-time systems to handle different requirements:

  • LiveKit (WebRTC): Handles heavy audio/video streaming via UDP/TCP.
  • SignalR (WebSockets): Handles lightweight control messages (Chat, Hand Raises, Whiteboard Sync).

5. System Analysis & Design

UML & Diagrams

Comprehensive Use Case Diagrams, Sequence Diagrams, and ERDs were crafted to ensure the team stayed aligned during the development lifecycle.

6. AI Models

Intelligence at the Core

MeetVerse isn't just a communication tool; it actively improves the communication medium using artificial intelligence.

6. AI Models

The Audio Shield Microservice

A standalone Python application built with FastAPI to ensure non-blocking, high-throughput asynchronous processing.

6. AI Models

PyTorch Noise Suppression

We integrated a Deep Learning model trained on massive datasets of human speech and environmental noise.

  • Accepts raw PCM audio streams via WebSocket.
  • Applies a mask to filter out non-human frequencies.
  • Returns cleaned audio with near-zero perceived latency.

6. AI Models

Deepgram Speech-to-Text

We bypassed standard transcription APIs in favor of Deepgram's neural network-based architecture.

  • Achieves sub-second transcription speeds.
  • Highly accurate punctuation and casing.

6. AI Models

Handling Race Conditions

AI processing introduces latency. We engineered robust state-management in React to ensure the UI gracefully handles AI connection delays, falling back to standard audio if the AI server is unreachable.

6. AI Models

Future AI Integration

The microservice architecture allows us to easily hot-swap models in the future, opening doors for Real-time Translation and meeting summarization.

7. Testing

Ensuring Reliability

A platform designed for enterprise communication cannot fail during critical moments. We implemented rigorous testing protocols.

7. Testing

Backend Unit Testing

We wrote comprehensive tests for the Application layer (Services) using xUnit and Moq.

  • Mocking the Database Context to test business logic in isolation.
  • Verifying JWT token generation and validation.

7. Testing

Integration & API Testing

Controllers were tested using Postman and Swagger.

  • Verified endpoint security (Authorize attributes).
  • Tested the upload flows to Google Drive for large files.
  • Handled 400/404/500 edge cases gracefully.

7. Testing

Frontend Component Testing

React components were rigorously tested for state memory leaks, especially surrounding the MediaRecorder and WebSocket listeners.

7. Testing

End-to-End QA Audit

Before release, we performed a deep-dive QA audit.

  • Resolved a critical race condition during initial meeting join.
  • Secured LiveKit tokens strictly behind authorized endpoints.
  • Removed hardcoded keys, migrating entirely to secure AppSettings.

9. Back-End

The Engine of MeetVerse

Robust, scalable, and heavily optimized ASP.NET Core 8 Web API.

9. Back-End

ASP.NET Identity & JWT

Authentication is handled securely. Passwords are hashed, and stateless JWTs are issued for high-performance API authorization.

9. Back-End

Dependency Injection

Our entire backend relies heavily on the built-in IoC container.

  • Services like GoogleDriveService and EmailService are injected where needed.
  • Promotes loose coupling and easier testing.

9. Back-End

Exception Handling Middleware

We implemented a global exception handler. Instead of crashing or leaking stack traces to the client, it returns standardized, predictable JSON error responses.

9. Back-End

Database Migrations

Entity Framework Core Code-First migrations allow us to track database schema changes iteratively alongside our Git commits, ensuring perfect sync between code and database.
